# Aggregates

## Organization

`Organization` is the tenant aggregate root. It owns API keys, report requests,
events, artifacts, audit logs, and retention policy through `organization_id`.

Important invariants:

- tenant slug is unique
- retention is tenant-scoped
- API keys cannot authorize another tenant

## Report

`Report` is the main workflow aggregate. It owns template, format, filters,
status, fingerprint, lifecycle timestamps, and terminal state.

Important invariants:

- status transitions are controlled by the Reports context
- duplicate creation is constrained by tenant plus idempotency key
- equivalent requests can be deduplicated by tenant plus fingerprint
- terminal reports can be retried only through explicit retry behavior

## Report Event

`ReportEvent` is an append-only timeline attached to a report. Events are
operator-facing facts, not a replacement for current report state.

Important invariants:

- every event belongs to one report and one organization
- payloads must not contain financial rows, API keys, credentials, or signed
  URLs
- event names follow the documented lifecycle contract

## Artifact

`Artifact` is the download metadata aggregate for generated bytes. PostgreSQL
stores metadata such as storage key, checksum, byte size, content type, and
expiry. The bytes live behind the configured artifact-storage adapter.

Important invariants:

- artifact ownership is tenant-scoped through report and organization metadata
- signed URLs must expire no later than artifact availability
- storage key is internal and not equivalent to public authorization
