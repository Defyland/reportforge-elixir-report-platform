# Large Report Pipeline

ReportForge treats large financial exports as an operational workflow, not as a
large HTTP response. The service accepts intent, persists lifecycle state,
delegates execution to a durable worker, writes artifacts through a storage
boundary, and exposes only short-lived download access.

## Goals

- Keep API request latency independent from report generation time.
- Keep worker memory bounded when report sizes grow.
- Preserve tenant isolation across request, execution, storage, and download.
- Make every important state transition inspectable through report events.
- Support local development now while keeping S3-compatible storage production-ready.

## Non-goals

- Do not implement new exporters in this phase.
- Do not introduce Kubernetes, Helm, or cluster-specific manifests yet.
- Do not introduce a data lake, CDC pipeline, or analytical warehouse layer.
- Do not make signed download URLs permanent integration identifiers.

## Pipeline

1. Request intake validates the tenant API key, payload shape, template, format,
   filters, and idempotency key.
2. Deduplication computes a tenant-scoped fingerprint and persists the report in
   PostgreSQL under unique constraints.
3. Queue handoff stores the report state and enqueues an Oban job in the same
   transactional workflow.
4. Worker execution transitions the report to `running` and emits progress
   events as work advances.
5. Artifact staging writes generated bytes through `ReportForge.ArtifactStorage`
   instead of coupling the report domain to a concrete storage backend.
6. Completion records checksum, byte size, row count, artifact metadata, and a
   signed download token with explicit expiration.
7. Retention cleanup removes expired metadata and storage objects according to
   tenant retention policy.

## Stream-first design constraints

- Exporters should process rows as streams or bounded chunks.
- Workers should not build full report bodies in memory for large datasets.
- Checksums and byte counts should be computed while bytes are written.
- Temporary files, when needed, must be scoped per report attempt and cleaned on
  failure.
- Storage writes should be retryable only when the failure is transient.
- Progress events should be useful for operators without exposing financial row
  contents.

## Current implementation boundary

The current generator is intentionally small and deterministic. It proves the
orchestration path, lifecycle state, idempotency, storage boundary, downloads,
and tests. Future large exporters should plug into this pipeline without
changing the API contract or report lifecycle semantics.

## Operational checks

- `/readyz` confirms database and runtime readiness.
- `/metrics` exposes HTTP, report, queue, and cleanup counters.
- OpenTelemetry trace context flows from the request into the worker.
- `scripts/smoke.sh` validates request, queue, storage, metrics, and download
  behavior in the production-like Compose stack.
